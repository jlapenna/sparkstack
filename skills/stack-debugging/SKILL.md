# Stack Debugging with Distributed Tracing

In modern, multi-host, orchestration-heavy environments (like `spark-stack`), debugging purely from container stdout logs can be blind to context and request flow. 

Distributed tracing (OpenTelemetry) provides an end-to-end visualization of the request lifecycle. It allows you to:
- Identify precisely where a request died or hung (e.g., `openclaw-gateway` -> `litellm` -> `vllm-main`).
- Inspect complete payloads (e.g., prompts and completions via `gen_ai.input.messages`).
- Uncover configuration failures (like missing DB connection keys) by tracing `error=true` span attributes back to the source container.

## 1. Tempo Architecture Overview

Traces are generated by SDKs in the applications (vLLM, LiteLLM, OpenClaw) and exported via OTLP (`OTEL_EXPORTER_OTLP_ENDPOINT=http://alloy:4318/v1/traces`). 
Grafana Alloy buffers and forwards these to Grafana Tempo (`http://localhost:3200`), where they are indexed and stored.

**Important Note on Asynchrony:** Trace ingestion and indexing are asynchronous. When debugging, you must wait ~10-15 seconds after an event for Tempo to reflect the trace in its search API.

## 2. Searching for Traces

Use the Tempo Search API to find recent traces. 

**Basic Recent Search (Last 15 traces):**
```bash
curl -s "http://localhost:3200/api/search?limit=15" | jq '.traces[] | .traceID'
```

**Filter by Service Name:**
Useful when a specific node (like `litellm`) is acting up:
```bash
curl -s "http://localhost:3200/api/search?limit=20&tags=service.name=litellm" | jq '.traces[] | .traceID'
```

**Filter by Endpoint/Target:**
Useful to find specific completions:
```bash
curl -s "http://localhost:3200/api/search?limit=20&tags=http.target=/api/v1/chat/completions" | jq '.traces[] | .traceID'
```

## 3. Fetching and Analyzing a Trace

Once you have a `traceID` (e.g., `7722917bbf566111293f51c2434aedaa`), you can pull the full OpenTelemetry JSON structure. 

### A. Verify End-to-End Linkage
To verify that all expected services successfully connected and propagated the trace context:
```bash
curl -s http://localhost:3200/api/traces/<traceID> | jq '[.batches[].resource.attributes[] | select(.key == "service.name") | .value.stringValue] | unique'
```
*Expected Output:* `["litellm", "openclaw-gateway", "vllm-main"]`
*Failure Indicator:* If a service is missing, the trace context was lost (e.g., missing `traceparent` headers) or the OTLP exporter failed in that container.

### B. Inspecting Payloads (Generative AI Inputs/Outputs)
To debug prompt truncation, parsing errors, or system instructions:
```bash
# Print the exact string value of the input messages
curl -s http://localhost:3200/api/traces/<traceID> | jq -r '[.batches[].scopeSpans[].spans[].attributes[]? | select(.key == "gen_ai.input.messages")] | .[0].value.stringValue'

# Check the length of the payload
curl -s http://localhost:3200/api/traces/<traceID> | jq -r '[.batches[].scopeSpans[].spans[].attributes[]? | select(.key == "gen_ai.input.messages")] | .[0].value.stringValue' | wc -c
```

## 4. Common Tracing Pitfalls & Fixes

### A. Truncated Span Attributes (The 2048 Limit)
If `gen_ai.input.messages` or other large fields are truncated exactly at 2048 characters, it is usually caused by the backend, not the application.
- **Tempo Storage Limit**: Tempo truncates attributes to `2048` bytes by default to prevent OOMs. To fix, configure `max_attribute_bytes: 131072` (or higher) under `distributor:` in `tempo.yaml`.
- **OTEL SDK Limit**: OTel SDKs also enforce limits. Ensure `OTEL_ATTRIBUTE_VALUE_LENGTH_LIMIT=131072` is injected into the container environments.

### B. Broken Trace Chains (Orphaned Spans)
If you see traces that contain `openclaw-gateway` but not `litellm` (or vice versa):
- Check if `LITELLM_FORWARD_TRACEPARENT_TO_LLM_PROVIDER=True` is enabled.
- Verify the HTTP client making the downstream request is passing the W3C `traceparent` header.

### C. Missing Traces Entirely
If a service is completely missing from Tempo:
- Verify network reachability: Ensure the container can resolve and route to `alloy:4318`.
- Ensure OTel is enabled (`VLLM_OTEL_TRACING_ENABLED=1`).
- Check Alloy logs (`docker compose logs alloy`) for `rpc error` or dropping metrics, which indicates backend ingestion failures.
